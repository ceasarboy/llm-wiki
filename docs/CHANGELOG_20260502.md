### 一、核心变更

#### 1. 增强版Prompt三步策略集成到主代码

**变更文件**：`scripts/agent_g.py`

**变更内容**：
- `generate_all_in_one()` 函数使用增强版三步策略Prompt：
  - 第一步：结构分析（识别论文类型、章节列表、核心推理链、必须覆盖的内容）
  - 第二步：内容生成（基于结构分析生成论文+实体+概念）
  - 第三步：覆盖完整性自检（逐项检查覆盖情况并评分）
- `generate_paper_page()` 函数同步更新为增强版Prompt

- 实体/概念解析函数 `_parse_entities()` 和 `_parse_concepts()` 增加按英文名去重逻辑

**测试结果**：
- 基线Prompt vs 增强版Prompt对比测试，增强版显著优于基线
- 增强版生成的方法章节更详细，实验结果更完整

#### 2. 实体/概念跨论文融合模块

**新建文件**：`scripts/merge.py`

**核心功能**：
- **4层匹配策略**识别重复实体/概念：
  1. 英文标题精确匹配
  2. 中文标题精确匹配
  3. 英文归一化匹配（去除非字母数字后比较）
  4. 模糊匹配（SequenceMatcher ≥ 0.85）
- **3级融合决策**：
  - 相似度 ≥ 0.9：仅追加相关论文链接，不修改内容
  - 0.5 ≤ 相似度 < 0.9：追加引用来源 + 相关论文链接
  - 相似度 < 0.5：创建新文件

**融合规则确认**（2026-05-02）：
| 场景 | 引用来源 | 相关论文 | 内容处理 |
|------|----------|----------|----------|
| 首次生成 | 生成文章 | 该文章 | 创建新文件 |
| 再次生成，内容无差异 | 不变 | 追加第二篇论文 | 不修改内容 |
| 再次生成，需增加内容 | 追加第二篇文章 | 追加第二篇论文 | 合并内容 |

### 四、经验教训

1. **先分析后生成**：LLM生成前先要求分析论文结构，显著提高内容覆盖度
2. **区分内容类型规则**：论文不需要Source ID，实体和概念需要，避免冗余标注
3. **后处理脚本要谨慎**：避免覆盖LLM已生成的丰富内容，检测已有链接时跳过
4. **审核评分要细粒度**：粗粒度检查导致评分无区分度，需要针对章节长度、数值结果等具体检查

## 第二次更新 - 搜索系统与前端修复

### 六、搜索系统升级

#### 1. 语义嵌入替换

**变更文件**：`scripts/indexer_simple.py`, `scripts/qmd_search_simple.py`

**变更内容**：
- `SimpleHashEmbedding`（MD5哈希伪嵌入）→ `SentenceTransformerEmbedding`（all-MiniLM-L6-v2 语义嵌入）
- 设置 `HF_HUB_OFFLINE=1` 和 `TRANSFORMERS_OFFLINE=1` 确保离线运行
- 使用单例模式避免重复加载模型

**问题根因**：SimpleHashEmbedding 使用MD5哈希生成向量，语义相似的文本向量之间没有任何关联，向量检索（权重0.7）形同虚设，实际只靠BM25关键词检索（权重0.3）工作。搜索"Transformer是什么"返回无相关结果。

#### 2. BM25中文分词改进

**变更文件**：`scripts/qmd_search_simple.py`

**变更内容**：
- 分词前先用正则分离中英文边界（`Transformer是什么` → `Transformer` + `是什么`）
- 中文部分生成 bigram/trigram 索引
- 扩充停用词列表（增加"什么"、"怎么"、"如何"等）

#### 3. 标题匹配增强

**变更文件**：`scripts/qmd_search_simple.py`

**变更内容**：
- 当查询词与页面标题有重叠时，对概念/实体页给予0.5基础提升，论文页给予0.2提升
- 按重叠比例计算最终提升分数
- 解决概念/实体页内容少导致排名低的问题

#### 4. 搜索范围扩大

**变更文件**：`scripts/qmd_search_simple.py`

**变更内容**：
- 向量检索和关键词检索的获取量从 `top_k*2` 增加到 `top_k*4`
- 确保标题匹配增强能覆盖到更多候选结果

**验证结果**：

| 查询 | 第1条结果 | 状态 |
|------|-----------|------|
| Transformer是什么 | Transformer (概念页, score=1.051) | ✅ |
| 注意力机制 | 缩放点积注意力 (概念页, score=0.926) | ✅ |

### 九、经验教训（第二次更新）

1. **搜索系统不能用伪嵌入**：SimpleHashEmbedding虽然接口兼容ChromaDB，但生成的向量无语义关联，向量检索完全无效。必须使用真正的语义嵌入模型。
2. **中文分词需要特殊处理**：中英文混合文本（如"Transformer是什么"）需要先分离中英文边界，否则整个字符串被当作一个token。
3. **概念/实体页需要标题匹配增强**：概念/实体页内容通常较少，纯向量相似度排名低，需要标题匹配提升排名。
4. **CORS配置要覆盖所有前端端口**：Vite开发服务器可能同时运行在5173和5174端口。
5. **401错误需要自动处理**：token过期时前端应自动跳转登录页，避免用户看到空白页面。
6. **字段名一致性**：SQLAlchemy模型定义的字段名（password_hash）必须与代码中使用的字段名一致。
