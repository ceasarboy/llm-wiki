# 文章录入流程设计文档 v2.0

## 一、流程概述

```
PDF文件 → Markdown转换 → LLM生成(论文+实体+概念) → Subagent审核 → 存档/融合
```

---

## 二、详细流程

### 阶段1：PDF转Markdown

**输入**：PDF文件目录 `raw/papers/pdf/`

**处理**：使用 marker-pdf 转换

**输出**：Markdown文件到 `raw/papers/markdown/`

**确认标准**：
- 文件大小 > 100字节
- 内容不为空

**现有程序**：`api/main.py` 第798-877行

**修改意见**：
- 增加转换后的自动确认步骤
- 记录转换状态到状态文件

---

### 阶段2：LLM生成

**输入**：Markdown文件

**处理**：LLM生成三种内容

**输出**：
1. 论文页面（wiki/papers/）
2. 实体页面（wiki/entities/）
3. 概念页面（wiki/concepts/）

**Prompt要求**：

#### 2.1 论文生成Prompt（增强版 v2.0）

采用"先分析后生成"的三步策略：

**第一步：结构分析**
```
分析论文完整结构，输出：
- 论文类型（常规论文/综述文章/技术报告）
- 章节列表（每章核心内容概述）
- 核心推理链（问题→方法→验证→结论）
- 必须覆盖的内容（≥5个技术点、≥3个实验结果）
```

**第二步：内容生成**
```
将论文转换为结构化的中文wiki页面，包含：
- 基本信息（arXiv ID、作者[中文音译-英文原名]、机构[中文名-英文名]）
- 核心观点（一句话概括，≤50字）
- 摘要（300-600字，覆盖核心贡献和技术路线）
- 问题定义（100-300字，说明背景和挑战）
- 方法（500-1200字，覆盖所有子方法，保留关键公式）
- 实验验证（列出所有实验任务和数据集，保留关键数值结果）
- 结论与展望（100-300字）
```

**第三步：覆盖完整性自检**
```
检查第一步中"必须覆盖的内容"是否全部覆盖
评分：内容覆盖/方法细节/实验结果 各X/10
```

**硬性要求**：
1. 所有内容使用中文
2. 概念提取数量必须≥3个
3. 方法章节必须覆盖原文所有子方法
4. 实验章节必须包含具体数值结果
5. 不要添加 [Source: ...] 标记（仅实体和概念需要）
6. 同一实体/概念不要重复提取

#### 2.2 实体生成Prompt

**实体定义**：具体、可命名的现实世界对象

**实体类型**：
| 类型 | 说明 | 示例 |
|------|------|------|
| person | 人物：作者、研究者 | Chelsea Finn, Sumita Mishra |
| organization | 机构：大学、实验室、公司 | Stanford, Google DeepMind |
| location | 地点：有上下文的地名 | campus locations |
| device | 设备：硬件平台、机器人 | Franka Emika Panda, PR2 Robot |
| dataset | 数据集：benchmark数据集 | D5RL, RLBench |

**输出格式**：
```yaml
---
title: 实体名称
type: entity
entity_type: person|organization|location|device|dataset
aliases: [别名1, 别名2]
tags: [标签1, 标签2]
source: [[raw/papers/markdown/文件名.md]]
created: 日期
updated: 日期
status: generated
confidence: high|medium|low
---
# 实体名称

## 概述
1-2句话定义该实体及其与论文的相关性。

## 关键信息
- **属性1**: 值（必须来自原文）
- **属性2**: 值

## 引用来源
- [论文标题](论文链接) - 该论文提及此实体。
[Source: 论文标题]

## 相关论文
- [[papers/论文文件名_论文]]
```

**边界判断规则**：
```
是实体：
- 人名、机构名、地名 → entity
- 具体设备型号、硬件平台 → entity
- 数据集名称 → entity

不是实体（是概念）：
- 算法/模型名称 → concept
- 方法论/框架名称 → concept
- 学习范式 → concept
- 技术思想 → concept
```

#### 2.3 概念生成Prompt

**概念定义**：抽象、可复用的想法/方法/理论

**概念类型**：
| 类型 | 说明 | 示例 |
|------|------|------|
| paper | 论文本身 | 作为研究贡献 |
| algorithm | 算法/模型 | MAML, BERT |
| methodology | 方法论 | Object-Level Adaptation |
| learning-paradigm | 学习范式 | meta-learning, few-shot learning |
| technical-idea | 技术思想 | domain randomization |
| evaluation-metric | 评估指标 | sample efficiency |

**输出格式**：
```yaml
---
title: 概念名称
type: concept
concept_type: paper|algorithm|methodology|learning-paradigm|technical-idea|evaluation-metric
tags: [标签1, 标签2]
source: [[raw/papers/markdown/文件名.md]]
created: 日期
updated: 日期
status: generated
confidence: 0.8
related_papers: []
llm_enhanced: true
---
# 概念名称

## 定义
一段精确、无歧义的定义段落。

## 核心特征
- **特征1**: 详细说明
- **特征2**: 详细说明

## 应用场景
- 场景1
- 场景2

## 引用来源
- [论文标题](论文链接) - 该论文提出或涉及此概念。
[Source: 论文标题]

## 相关论文
- [[papers/论文文件名_论文]]
```

**边界判断规则**：
```
是概念：
- 算法/模型名称 → concept
- 方法论/框架名称 → concept
- 学习范式 → concept
- 技术思想 → concept
- 评估指标 → concept

不是概念（是实体）：
- 作者名、机构名 → entity
- 硬件平台、机器人 → entity
```

**现有程序**：`scripts/agent_g.py`

**修改意见**：
- 当前只生成论文，需要增加实体和概念的独立生成
- 参考 `mdtest/agents/generator.py` 的Prompt设计
- 增加边界判断示例表

### 阶段3：程序审核（Agent-R）

**审核方式**：程序规则打分（非LLM审核），审核不通过时调用LLM修正

**审核对象**：论文、实体、概念

**审核维度**：
| 维度 | 权重 | 说明 |
|------|------|------|
| 完整性 | 25% | 关键章节是否存在 |
| 准确性 | 25% | Source ID比例 |
| 结构规范性 | 20% | Frontmatter格式 |
| 可发现性 | 15% | 交叉引用 |
| 冲突处理 | 15% | Conflict标记 |

**通过标准**：综合分 ≥ 7.5

**审核流程**：
```
第1轮生成 → 审核
  ├─ ≥7.5 → 通过 → 存档
  └─ <7.5 → 第2轮修改 → 审核
              ├─ ≥7.5 → 通过 → 存档
              └─ <7.5 → 存档，标注"待完善"
```

**现有程序**：`scripts/review.py`

**修改意见**：
- 当前只审核论文，需要增加实体和概念的审核
- 审核最多2轮，不是当前的无限重试
- 第2轮不通过标注"待完善"而非"人工审核"

---

### 阶段4：实体/概念融合

**触发条件**：生成的实体或概念在系统中已存在

**重复识别**（4层匹配策略）：
| 层级 | 方式 | 示例 |
|------|------|------|
| 1 | 英文标题精确匹配 | `Lei Li` = `lei li` |
| 2 | 中文标题精确匹配 | `李磊` = `李磊` |
| 3 | 英文归一化匹配 | `Knowledge Distillation` = `knowledgedistillation` |
| 4 | 模糊匹配（SequenceMatcher ≥ 0.85） | `Geoffrey Hinton` ≈ `geoffrey e. hinton` |

**融合决策**（基于内容相似度）：
| 相似度 | 引用来源 | 相关论文 | 内容处理 |
|--------|----------|----------|----------|
| ≥ 0.9 | 不变 | 追加第二篇论文 | 不修改内容 |
| 0.5 ~ 0.9 | 追加第二篇文章 | 追加第二篇论文 | 合并内容 |
| < 0.5 | 新文章 | 新文章 | 创建新文件 |

**现有程序**：`scripts/merge.py`

## 七、新增功能需求

### 7.1 论文复审功能

**触发方式**：前端"复审"按钮

**处理流程**：
```
点击复审 → 发送原文+转换后文件给LLM → LLM判断是否需要修改
    ├─ 不需要修改 → 提示用户
    └─ 需要修改 → 进入修改流程
                    ├─ 内容修改 → subagent审核 → 通过后存档
                    ├─ 实体新增/修改 → subagent审核 → 通过后存档
                    └─ 概念新增/修改 → subagent审核 → 通过后存档
```

### 7.2 版本控制

**目录结构**：
```
wiki/
├── papers/
│   └── xxx_论文.md          # 当前版本
├── entities/
│   └── xxx.md               # 当前版本
├── concepts/
│   └── xxx.md               # 当前版本
└── history/                 # 历史版本目录
    ├── papers/
    │   ├── xxx_论文_v1.md
    │   └── xxx_论文_v2.md
    ├── entities/
    └── concepts/
```

**版本命名规则**：`{原文件名}_v{版本号}.md`

**日志记录**：
```yaml
- timestamp: 2026-04-26 12:00:00
  action: revise
  page: papers/xxx_论文.md
  old_version: history/papers/xxx_论文_v1.md
  new_version: papers/xxx_论文.md
  reason: 复审修改
  review_score: 8.5
```

### 7.3 人工审核状态

**状态字段**：
```yaml
status: generated|reviewed|stable|requires_manual_review
manual_review: pending|approved  # 新增字段
```

**前端显示**：
- 待审核状态：显示"人工审核"按钮（仅管理员可见）
- 已审核状态：显示"已人工审核"标签

**API端点**：
```
POST /api/pages/{id}/manual-review
```

### 7.4 待完善状态显示

**前端样式**：
```css
.status-needs-improvement {
  color: red;
  font-weight: bold;
}
```

**显示内容**：
- 红色粗体"待完善"标签
- 显示上次审核不通过的意见
- 完善后删除意见，状态更新

## 十三、经验教训

### 13.1 Prompt 设计经验

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 实体关键信息是占位符 | Prompt 使用 `[属性: 值]` 格式，LLM 直接复制 | 使用具体示例替代占位符 |
| 实体标题没有中文翻译 | Prompt 没有明确要求翻译人名 | 添加明确的格式要求和示例 |
| 论文标题使用期刊名 | 原始文档 frontmatter 的 title 字段错误 | 增加期刊名特征过滤逻辑 |
| 论文生成内容不覆盖全文 | 直接生成缺少全局视角 | 先分析结构再生成，三步策略 |
| 论文不应包含Source ID | 早期设计要求所有内容都标注来源 | 论文不标注，仅实体和概念需要 |
| 实体/概念重复生成 | 没有跨论文去重机制 | 实现4层匹配+3级融合策略 |

### 13.2 正则表达式问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 无法提取多行关键信息 | 正则 `.+` 默认不跨行 | 使用 `re.DOTALL` 标志 |
| 提取内容被截断 | 正则的贪婪匹配 | 使用非贪婪匹配 `.+?` 配合终止条件 |

### 13.3 文件名匹配问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 融合时找不到已存在文件 | 文件名格式不一致（连字符 vs 下划线） | 增加多种文件名格式候选 |

### 13.4 前端渲染经验

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 图片路径无法显示 | 路径映射不完整 | 增强 renderMarkdown 支持多种路径格式 |
| Markdown 字体太小 | 使用13px默认字体 | 调整为16px+行高1.8 |
| 自定义正则解析不稳定 | Markdown语法复杂，正则难以覆盖 | 引入 marked 库替代正则解析 |
| Wiki链接显示为占位符 | marked库处理后替换占位符失败 | 在marked解析前直接转换为HTML |
| 概念链接显示为WIKILINK_N | 前端未解析Wiki链接语法 | 增加实体/概念标签卡片组件 |

### 13.5 审核系统经验

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 审核评分区分度不足 | 检查项过于粗粒度 | 增加方法章节长度、实验数值、概述质量等细粒度检查 |
| postprocess_links覆盖LLM内容 | 后处理脚本无条件替换链接 | 检测已有`[[entities/`链接时跳过 |

### 13.6 最佳实践

1. **Prompt 示例要具体**：不要用占位符，用真实示例
2. **格式要求要明确**：列出具体格式要求和示例
3. **正则要考虑多行**：使用 `re.DOTALL` 和非贪婪匹配
4. **文件名要统一**：建立统一的文件名生成规则
5. **测试要全面**：测试不同类型的文档和边界情况
6. **不要重复造轮子**：Markdown等成熟领域使用开源库
7. **先分析后生成**：LLM生成前先要求分析结构，提高内容覆盖度
8. **区分内容类型的规则**：论文不需要Source ID，实体和概念需要
9. **后处理脚本要谨慎**：避免覆盖LLM已生成的丰富内容
10. **搜索系统必须使用真正的语义嵌入**：SimpleHashEmbedding等伪嵌入无法实现语义匹配，向量检索形同虚设
11. **中文分词需要特殊处理**：中英文混合文本需要先分离边界再分词
12. **概念/实体页需要标题匹配增强**：内容少导致纯向量排名低
13. **CORS配置要覆盖所有前端端口**：Vite可能同时运行在5173和5174
14. **401错误需要自动处理**：token过期时前端应自动跳转登录页
