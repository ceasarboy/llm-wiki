## 二、架构审查

### 2.1 整体架构

```
[PDF上传] → [Marker PDF→MD] → [Raw MD存储]
                                    ↓
[用户搜索] ← [ChromaDB向量库] ← [Indexer索引]
                                    ↑
[Wiki页面] ← [Agent_G生成] ← [Batch编排]
                 ↓              ↑
           [Agent_R审核] → [Fix Cycle修复]
                 ↓
           [Merge融合] → [实体/概念去重合并]
```

### 2.2 优点

1. 管道式处理模型契合 LLM-Wiki 的"编译"理念
2. 审核修复循环 (review_with_fix_cycle) 设计有深度
3. 实体/概念融合机制的多层匹配策略得当
4. 异步任务模式处理长耗时操作正确

### 2.3 待改进项（进入 V2 计划）

1. **核心管道耦合在脚本层** - batch.py 通过 subprocess 调用，应模块化为直接函数调用
2. **VaultIndex 是静态快照** - 文件变更后需手动 rescan，应增量更新
3. **ChromaDB 与 Wiki 文件双写** - 存在不一致风险

## 四、代码质量审查

### 4.1 亮点

- `_safe_path` / `_safe_filename` 路径防护设计得当
- `_clean_title` 函数处理了 PDF 转 MD 的标题污染问题
- 融合机制的多层匹配策略效率-准确性权衡好
- `atomic_write` 简洁有效

### 4.2 待改进项（进入 V2 计划）

1. **main.py 过于庞大 (~2000行)** - 应拆分为多个 Router 模块
2. **前端 token 提取逻辑重复** - 应抽取 getToken() 工具函数
3. **call_llm 缺乏重试机制** - 应增加指数退避重试
4. **代码风格不一致** - datetime、配置加载方式不统一

---

## 五、安全性审查

### 5.1 已正确实现
- 路径遍历防护
- JWT 认证 + 角色授权
- 生产环境密钥强制更换
- 登录速率限制 (5次/分钟)
- 注册防枚举（统一错误消息）
- 密码复杂度校验
- CORS 白名单
- XSS 防护 (DOMPurify)

### 5.2 待改进项（进入 V2 计划）

1. **Token 存 localStorage** - 生产环境应用 httpOnly cookie
2. **缺少请求体大小限制** - 应添加 MAX_UPLOAD_SIZE
3. **密码在日志中可能泄露** - 需确保日志过滤器屏蔽敏感字段

## 六、可靠性与容错

### 6.1 待改进项（进入 V2 计划）

## 九、总结

设计理念清晰、技术选型精良、核心功能完整。管道式处理 + 审核修复循环 + 智能融合去重体现了对 LLM-Wiki 理念的深入理解。

V1 版本已补齐了系统架构图、API 业务文档和部署运维文档，项目文档完整度从"基础可用"提升到了"专业化"水平。

弱点集中在工程化程度不足——代码组织偏脚本化、模块边界模糊、缺少系统级的容错和自愈机制。但考虑到项目规模和目标用户数（≤10人），不影响核心功能使用。

**一句话总结**: 架构思路正确，文档已完善，工程实现需要从"能跑"进化到"好维护"。
